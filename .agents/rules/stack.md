# Scholar Management System — Tech Stack

---

## Overview

The system is split into three separate applications sharing one backend API and one database.

```
Scholar web app (React PWA)
Evaluator desktop app (PySide6)
        ↓
FastAPI backend (Python)
        ↓
Supabase (PostgreSQL + Storage)
```

---

## Scholar web app

**Framework:** React 18  
**Build tool:** Vite  
**Type:** Progressive Web App (PWA)  
**Deployment:** Vercel or Netlify (free tier)  
**Runs on:** Any browser — desktop, mobile, tablet

| Package | Purpose |
|---|---|
| `react` | UI framework |
| `react-router-dom` | Client-side routing |
| `axios` | HTTP requests to FastAPI backend |
| `@vitejs/plugin-pwa` | Service worker, offline caching, installable PWA |
| `vite` | Build tool and dev server |

**Why React + Vite:**  
Vite is fast to set up and the ecosystem is large. The PWA plugin handles offline caching with a service worker, which satisfies the offline requirement for scholars without building a complex sync engine.

**Offline strategy:**  
The service worker caches the app shell and the scholar's own data (profile, grades, allowance history) for read access. Write operations (submitting changes) queue locally and fire when connectivity returns.

## Evaluator desktop app

**Framework:** PySide6 (Qt6 bindings for Python)  
**Platform:** Windows only  
**Distribution:** PyInstaller executable (.exe)  
**Runs on:** Office workstations

| Package | Purpose |
|---|---|
| `PySide6` | Desktop UI framework (Qt6) |
| `requests` | HTTP requests to FastAPI backend |
| `python-dotenv` | Load environment variables |

**Why PySide6 over PyQt6:**  
PySide6 is the official Qt binding maintained by The Qt Company and is LGPL licensed — safe for internal commercial use without purchasing a license. PyQt6 requires a commercial license for proprietary software. The APIs are nearly identical.

**Why a desktop app for evaluators:**  
Evaluators are office-based on Windows workstations. A desktop app gives richer UI for document previewing, bulk actions, report generation, and complex filtering — all of which are awkward in a browser for power users.

## Backend

**Framework:** FastAPI (Python)  
**Server:** Uvicorn (ASGI)  
**Deployment:** Railway or Render (free tier for development, paid for production)  
**Runs on:** Any Linux server or cloud platform

| Package | Purpose |
|---|---|
| `fastapi` | Web framework — routes, validation, OpenAPI docs |
| `uvicorn[standard]` | ASGI server |
| `sqlalchemy` | ORM — Python classes mapped to database tables |
| `alembic` | Database migrations |
| `psycopg2-binary` | PostgreSQL driver |
| `python-jose[cryptography]` | JWT creation and verification |
| `passlib[bcrypt]` | Password hashing (bcrypt) |
| `bcrypt==3.2.2` | Pinned — passlib is incompatible with bcrypt 4.x |
| `passlib==1.7.4` | Pinned — last version with stable bcrypt support |
| `python-multipart` | File upload support |
| `supabase` | Supabase Storage client (file upload, signed URLs) |
| `google-auth` | Google ID token verification for OAuth |
| `python-dotenv` | Load `.env` file |

**Why FastAPI:**  
FastAPI is the fastest Python web framework for building APIs. It auto-generates interactive `/docs` via OpenAPI, has built-in request validation via Pydantic, and handles async file uploads natively. It's also the same language as the evaluator desktop app (Python), which reduces context switching.

**Pinned dependency note:**  
`bcrypt` must be pinned to `3.2.2` and `passlib` to `1.7.4`. Newer versions of `bcrypt` are incompatible with passlib's hashing interface and cause a `ValueError` on login. Always add these to `requirements.txt`.

## Database

**Provider:** Supabase (hosted PostgreSQL)  
**Database:** PostgreSQL 17  
**ORM:** SQLAlchemy 2.x  
**Migrations:** Alembic  
**Connection:** Session pooler (IPv4 compatible) on port 5432

**Why Supabase:**  
Supabase is a hosted PostgreSQL service with a dashboard, built-in file storage, and an auth system. It removes the need to manage a local PostgreSQL installation and is accessible from anywhere — the database is live on the internet from day one. The free tier covers development comfortably (500MB database, 1GB storage).

**Connection string format:**
```
postgresql://postgres.{project_ref}:{password}@aws-0-{region}.pooler.supabase.com:5432/postgres?sslmode=require
```

Use the **Session pooler** URL (not the direct URL) for IPv4 compatibility.

---

## File storage

**Provider:** Supabase Storage  
**Bucket:** `scholar-documents` (private)  
**Max file size:** 1MB  
**Allowed types:** JPEG, PNG, PDF  
**Access:** Signed URLs (120 second expiry) generated by FastAPI backend

**Why Supabase Storage:**  
Already included in the Supabase project — no separate account or billing. Private buckets mean files are never publicly accessible. Signed URLs mean access is always gated through the backend auth check.

**Storage path format:**
```
scholars/{scholar_id}/{doc_type}/{uuid}_{original_filename}
```

## Authentication

**Scholar auth:** Google OAuth 2.0  
**Evaluator auth:** Email + password (bcrypt hashed)  
**Token format:** JWT (HS256, 8-hour expiry)  
**Token storage:** Browser localStorage (scholar web), in-memory (evaluator desktop)

**Google OAuth flow:**
1. Scholar web app handles Google login popup
2. Google returns a Google ID token (not an access token)
3. Web app sends the ID token to `POST /auth/google`
4. Backend verifies it with Google's public keys using `google-auth`
5. Backend checks the extracted email against the `users` table
6. If registered as a scholar, a system JWT is issued

**Why separate auth for evaluators:**  
Evaluators are internal staff whose accounts are created manually by admins. They don't need OAuth — a simple email + password is more appropriate and avoids requiring evaluators to have a Google account.

## Environment variables

All sensitive config lives in `backend/.env` — never committed to git.

```env
DATABASE_URL=postgresql://postgres.{ref}:{password}@{host}:5432/postgres?sslmode=require
JWT_SECRET=your-long-random-secret
SUPABASE_URL=https://{project_ref}.supabase.co
SUPABASE_SERVICE_KEY=eyJ...
GOOGLE_CLIENT_ID=your-google-client-id.apps.googleusercontent.com
ENV=development
```

A `backend/.env.example` with empty values is committed to git as a reference for other developers.

| Component | Development | Production |
|---|---|---|
